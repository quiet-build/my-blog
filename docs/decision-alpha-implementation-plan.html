Decision Alpha

A skill-verification tool for copy trading. It watches eToro Popular Investors over time, grades their actual buy/sell decisions, and tells you who is genuinely skilled — before you copy them.

0Overview

BullAware (now eToro-owned) and BigFatBat already analyse returns and current holdings. Nobody grades the quality of an investor's decisions over time — because eToro's API exposes no public ledger of an arbitrary investor's closed trades (only the authenticated user's own — see §2), so the data has to be accumulated by snapshotting. That accumulation is the moat. This plan builds the data collector first, launches on an instant-value feature that needs no history, and lets the decision-grading engine mature underneath into the retention/paid layer.

One-line pitch: "Returns tell you what happened. Decision Alpha tells you whether it was skill — so you know who's actually worth copying."

2The core constraint (verified against the API)

There is no public endpoint that returns an arbitrary investor's closed-trade ledger. GET /user-info/people/{username}/tradeinfo returns aggregate stats only; GET /user-info/people/{username}/portfolio/live gives only currently open positions. (The authenticated user's own real-account history does exist via GET /trading/info/trade/history with openRate/closeRate + timestamps — usable for personalization & the user's own copying, but not for grading other PIs.)

Consequences that shape the whole build:

  • Buys are directly observable — a live position carries openRate (entry price), openTimestamp, instrumentId, isBuy, leverage, investmentPct, netProfit.
  • Sells are observable only by inference — a position that disappears between two snapshots = an exit. Exit timing is known to ~1 day; exit price is estimated from market candles.
  • No backfill. We only know decisions made after we start polling. → start the collector immediately (see §9).
  • Copy-trades are separable — positions carry socialTradeId / parentPositionId, so mirrored/mechanical trades can be excluded and only genuine discretionary decisions graded.

3Architecture

① Universe builder — /user-info/people/search + /pi-data/copiers → the set of PIs to track
↓ daily
② Snapshot collector (cron) — /portfolio/live, /gain, /tradeinfo per investor → raw snapshots
↓
③ Diff engine — compare consecutive snapshots → BUY / SELL events (ADD/TRIM backlog-only); copy-trades flagged
↓
④ Price enricher — /market-data/.../history/candles → instrument & sector prices around each event
↓
⑤ Scoring engine — Track-Record X-Ray, entry alpha, exit alpha, decision score, blow-up risk
↓
⑥ API + Web app (Next.js) — scorecards, screener, alerts, shareable cards

Steps ①–⑤ are a backend worker pipeline that runs on a schedule. Step ⑥ reads pre-computed results — the web tier never calls eToro on the user's critical path.

6Diff & event engine

For each investor, compare snapshot S(t-1) → S(t) by position_id:

ChangeEventCaptured
position_id appearsBUYexact open_rate, open_ts, instrument, size, leverage
position_id goneSELLinstrument, exit window; exit_price_est from candles at detected_at
investment_pct ↑ (same position)ADD?suspected — see caveat below
investment_pct ↓ (still open)TRIM?suspected — see caveat below
First snapshot = baseline, not buys. The initial crawl for each PI seeds its open positions as the starting state and emits no events. Events are derived only from the 2nd snapshot onward — otherwise every pre-existing position would be misclassified as a fresh BUY and pollute entry-alpha from day one. first_tracked_at marks the boundary; scores count only events after it.
Copy-noise filter: if social_trade_id is set (mirrored trade) or the position is part of a copied portfolio, mark is_copy = true and exclude from decision scoring. We grade only what the investor chose themselves.
ADD/TRIM is unreliable in v1 — ship BUY/SELL only. The public live-portfolio endpoint exposes investmentPct (portfolio-relative) but not absolute units/size. Market moves or equity changes shift investmentPct with no trade, so a pct delta ≠ a decision. v1 classifies only new positionId (BUY) and disappeared positionId (SELL). ADD/TRIM stays "suspected" until validated against real samples (and only if an absolute units field proves available).
Sampling caveat to log honestly: daily polling misses intraday round-trips (open + close between two snapshots). We surface scores as "based on observed decisions since {first_tracked_at}" and never imply completeness.

7Scoring engine

7.1 Track-Record X-Ray no accumulation needed — v1 launch feature

Correction (code review): the earlier draft launched on a "copier-gain gap" from copiersGain. That field denotes change in the number of copiers, not copiers' investment returns — and no public endpoint exposes an arbitrary investor's copier returns (CopyRealizedEquity_pnl exists only for your own copiers, authenticated). That wedge is removed. v1 instead launches on a forensic verdict computed from the public return series, which needs zero accumulation:

sharpe = mean(r) / stdev(r) · √periods  |  max_drawdown from cumulative gain  |   down_capture = mean(r | market<0)  |  consistency = % profitable months

Inputs: /user-info/people/{username}/gain (monthly/yearly) + /daily-gain + one /portfolio/live snapshot (current concentration & leverage). Output: a "real-skill vs. luck" verdict — risk-adjusted return, return-series drawdown, behaviour in down markets, consistency, and current portfolio risk. All available day one.

Drawdown caveat: /gain + /daily-gain give a return series, so drawdown here is return-series drawdown — not necessarily account-equity or intraday realized drawdown. Label it as such until the exact series semantics are verified with a provisioned key.

v1 must be opinionated, not a dashboard. The X-Ray's value is the verdict + explanation, not the raw metrics. Each scorecard leads with a plain-language call and the reason:
  • "Great headline return — but poor downside behaviour: −38% in the last drawdown, slow to recover."
  • "High gain is mostly one regime — 71% of the 2-yr return came from a single 4-month window."
  • "Consistent low-drama compounder — shallow drawdowns, 78% green months."
Simple verdicts + strong explanations + shareable cards. If it reads like "yet another risk-adjusted ranking," it loses to BullAware. This is the difference between v1 working and v1 being noise.
Honest positioning: this overlaps with eToro-owned BullAware. v1 X-Ray is the acquisition funnel; the moat is the Decision Alpha layer (§7.2–7.4) that accrues from snapshots and BullAware doesn't have. Win v1 on a sharper, opinionated verdict + speed; win retention on decisions-over-time.

7.2 Entry-timing alpha

For each BUY on instrument i at t₀, over horizon h ∈ {1w, 1m, 3m}:

entry_alpha(i,h) = return(i, t₀→t₀+h) − return(sector(i), t₀→t₀+h)

Investor entry alpha = investment_pct-weighted mean across evaluable buys. Positive = their buy timing beat just owning the sector.

7.3 Exit-timing alpha

For each SELL on instrument i at t₁, measure what happened after they left:

exit_alpha(i,h) = −[ return(i, t₁→t₁+h) − return(sector(i), t₁→t₁+h) ]

Stock fell after the sale (relative to sector) → positive (good exit). Stock kept rising → negative (the "sold too early" case the user described).

Don't ship the naïve version. "Price rose after sale = bad sell" mislabels good managers who rotated capital. v2 upgrades exit_alpha to a reallocation comparison: the sold stock's return vs. what they bought with the proceeds in the same window. Note: entry/exit alpha are v2 features — they need accrued snapshot history. v1 ships only the Track-Record X-Ray (§7.1). Until enough history accrues, exit alpha is labelled "directional, single-horizon."

7.4 Decision score & confidence

decision_score = w₁·z(entry_alpha) + w₂·z(exit_alpha)    confidence = f(number of evaluable decisions, tracking duration)

Z-scored across the tracked universe so the score is relative ("top 8% of PIs by entry timing"). Always shown with confidence — a 3-week-tracked investor is flagged "insufficient history," never given a false grade.

7.5 Blow-up risk v3 alerts

blowup_risk = z(↑riskScore trend) + z(weeklyDd) + z(exposure) + z(↓copiers / baseLineCopiers)

Rising risk + drawdown + leverage + copiers leaving → early-warning signal that drives the alert product.

8Phased roadmap

M0 Week 0–2 · private
Proof, not product. DO THIS FIRST Private collector on 20–50 PIs, daily snapshots → diff → decision events, plus a prototype scorer in a notebook. Validate three things before any UI: (1) snapshots actually surface interesting decision events; (2) scoring produces non-obvious rankings — not just a re-sort of return/risk; (3) the output is genuinely more compelling than plain return/risk metrics. Needs only one API key; low terms exposure. Gate: proceed only if the proof is convincing.
Gate before scaling
Confirm platform terms. Before scaling the collector or building UI, resolve: does eToro permit server-side aggregation of public PI data under one operator account, and does App Store distribution allow this monetization model? These can decide viability — treat as blocking, not paperwork-later.
v0 Week 2–3
Scale the collector. Expand to the top ~300–500 PIs; instrument→sector map + candle cache. Banks permanent history while v1 is built. (Only after M0 proof + terms gate pass.)
v1 Week 3–5
Launch — Track-Record X-Ray + screener. Zero cold-start (computed from the public return series). Per-investor page: risk-adjusted return, return-series drawdown, down-market behaviour, consistency, current concentration/leverage, shareable card. No end-user auth for public scorecards (served from the pre-crawled DB); auth is deferred to v3 personalization (see §4). The user-acquisition wedge.
v2 Month 2–3
Decision Alpha goes live. Now backed by real accumulated history: entry alpha, exit alpha (reallocation-aware), decision score + confidence, "best/worst calls" feed with charts. The retention layer.
v3 Month 3+
Alerts & personalization. Blow-up-risk + decision alerts on investors you copy, look-through exposure across your copies. Daily re-engagement → the paid tier.

9Cold-start strategy

The plan is designed so you never wait on data to launch. v1 ships on the Track-Record X-Ray (computed from the public return series — instant). Meanwhile the v0 collector banks the snapshot history that v2's Decision Alpha needs. By the time you launch v1, weeks of decision data are already accruing — a moat competitors can't backfill.

10Tech stack & scale

Recommended stack

  • Frontend/API: Next.js (App Router) — you already know it
  • DB: Postgres (Neon / Supabase)
  • ORM: Drizzle
  • Collector: dedicated Node worker (Railway / Fly) — not serverless; it needs a long-running scheduled loop with rate-limit pacing
  • Queue/cache: Redis (rate-limit token bucket + job dedupe)
  • Charts: lightweight (visx / Recharts)

Rate-limit budget (per user key)

Reads 60/min; tradeinfo-class 20/min. Daily crawl, 500 PIs:

CallCountClass
portfolio/live50060/min
gain50060/min
tradeinfo × periods500×N20/min
people/search (paged)~2060/min
candles (event-driven, cached)variable60/min
instrument metadata (cached)~once60/min

tradeinfo is the bottleneck: 500×N at 20/min = 25×N min/day. Legitimate mitigations: fetch tradeinfo weekly (not daily), single period, narrow the universe, cache aggressively. The crawler authenticates with the operator account's x-api-key + x-user-key (note: x-user-key is user-specific, not an app-level credential) — and only if eToro permits server-side public-PI aggregation under that account. Never end-users' keys (those are §4 context ②). Token-bucket + exponential backoff on 429.

Do not multi-key to scale. Limits are per user key, but provisioning multiple keys to raise an aggregate crawl ceiling reads as rate-limit circumvention — a terms/approval item, not a mitigation. If 500 PIs × daily won't fit one key's budget, request elevated/commercial limits from eToro; don't engineer around the cap.
